## Критические нарушения

### [CQRS-003] `*ReadRepositoryInterface` объявлены в Domain вместо Application

**Правило:** `*ReadRepositoryInterface` объявляется в **Application**-слое.  
**Эталон:** `src/Task/Application/Repository/TaskReadRepositoryInterface.php`

Все три Read-интерфейса Requirement-модуля размещены в `Domain\Repository\`:

```
src/Requirement/Domain/Repository/BusinessRequirementReadRepositoryInterface.php   ← должно быть в Application
src/Requirement/Domain/Repository/FunctionalRequirementReadRepositoryInterface.php  ← должно быть в Application
src/Requirement/Domain/Repository/NonFunctionalRequirementReadRepositoryInterface.php ← должно быть в Application
```

**Как исправить:** перенести в `src/Requirement/Application/Repository/`, обновить импорты в UseCase'ах и реализациях.

---

### [CQRS-003] `*Detail`-классы (read-DTO) размещены в `Domain\Model` вместо `Application\Dto`

**Правило:** read-DTO (детальные представления для запросов) принадлежат Application-слою.  
**Эталон:** `src/Task/Application/Dto/TaskDetail.php`, `TaskSummary.php`

Нарушители:

```
src/Requirement/Domain/Model/BusinessRequirementDetail.php   ← должно быть в Application\Dto
src/Requirement/Domain/Model/FunctionalRequirementDetail.php  ← должно быть в Application\Dto
src/Requirement/Domain/Model/NonFunctionalRequirementDetail.php ← должно быть в Application\Dto
```

Доменная модель (`Domain\Model`) должна содержать только агрегаты и Value Objects с бизнес-поведением. `*Detail` — это проекции для чтения, они не несут инвариантов и принадлежат Application.

**Как исправить:** перенести в `src/Requirement/Application/Dto/`, обновить импорты в Output-классах, Read-репозиториях и их реализациях.

### [CA] Write-репозитории принимают скалярные аргументы вместо доменной модели

**Правило (clean-architecture.md):** «Методы `create` и `update` принимают **доменную модель целиком** — передавать отдельные скалярные аргументы, DTO или массивы **запрещено**.»

Нарушители — все шесть write-интерфейсов и их реализации:

```php
// BusinessRequirementWriteRepositoryInterface — нарушение
public function create(int $projectId, string $code, string $description): BusinessRequirement;
public function update(int $id, string $description): void;

// FunctionalRequirementWriteRepositoryInterface — нарушение
public function create(int $projectId, string $code, string $description): FunctionalRequirement;
public function update(int $id, string $description): void;

// NonFunctionalRequirementWriteRepositoryInterface — нарушение
public function create(int $projectId, string $code, NonFunctionalRequirementType $type, string $description, ?string $acceptanceCriteria): NonFunctionalRequirement;
public function update(int $id, ?string $description, ?NonFunctionalRequirementType $type, ?string $acceptanceCriteria): void;
```

**Как исправить:** ввести доменные модели-агрегаты (или Value Objects) для каждого типа требования, передавать их целиком. Пример:

```php
// Правильно
public function create(BusinessRequirement $requirement): BusinessRequirement;
public function update(BusinessRequirement $requirement): void;
```

### [CA] Write-репозиторий персистирует два агрегата

**Правило (clean-architecture.md):** «Репозиторий персистирует ровно один агрегат. Всё необходимое передаётся в него как готовый аргумент из UseCase.»

Все три Write-реализации внутри `create()` делают два INSERT: в основную таблицу требований **и** в `core.project_entities` (чужой агрегат):

```php
// PdoBusinessRequirementWriteRepository::create()
// 1. INSERT INTO core.business_requirements ...
// 2. INSERT INTO core.project_entities ... ← персистирует чужой агрегат
```

То же в `PdoFunctionalRequirementWriteRepository` и `PdoNonFunctionalRequirementWriteRepository`.

**Последствия:**
- Нарушена граница ответственности репозитория
- Два INSERT выполняются без транзакции — при сбое второго требование будет создано без привязки к проекту

**Как исправить:** вынести вставку в `project_entities` в UseCase, делегировав её через `ProjectEntityWriteRepositoryInterface` (уже существует в `src/Task/Domain/Repository/ProjectEntityWriteRepositoryInterface.php`), и обернуть оба вызова в транзакцию.

## Замечания (не блокируют мерж, но стоит устранить)

### [CA] Кроссмодульная зависимость Requirement → Task на уровне Application

`GetFunctionalRequirementUseCase` (модуль `Requirement`) зависит на `TaskReadRepositoryInterface` из `Task\Application\Repository`.

```php
// src/Requirement/Application/UseCase/GetFunctionalRequirement/GetFunctionalRequirementUseCase.php
use App\Task\Application\Repository\TaskReadRepositoryInterface;
```

Правило гласит «межмодульные зависимости — только через Domain-интерфейсы». Это Application-to-Application зависимость через границу модуля. В текущей архитектуре обойти её сложно (Task не имеет Domain-level read repo), поэтому это допустимо как осознанное исключение, но его стоит задокументировать в CLAUDE.md или ADR.

---

### `UpdateFunctionalRequirementInput` и `UpdateBusinessRequirementInput` не поддерживают partial update

`description` объявлен как обязательный `string`, а не `?string`. Если в будущем у БТ/ФТ появятся дополнительные поля, потребуется переработка сигнатуры. `UpdateNonFunctionalRequirementInput` для сравнения использует nullable поля. Согласованность стоит обеспечить заранее.
